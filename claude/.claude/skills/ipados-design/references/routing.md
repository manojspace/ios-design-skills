# Routing

Load the smallest reference set that protects iPad quality.

## Common Paths

- Adapting an existing iPhone screen to iPad: read `product-parity.md`, `adaptive-layout.md`, and the relevant part of `components.md`.
- New iPad screen: read `product-parity.md`, `adaptive-layout.md`, `tokens.md`, and `components.md`.
- Sidebar, split view, inspector, windowing, or menu bar work: read `navigation-multitasking.md`.
- Pointer, trackpad, keyboard, Pencil, drag/drop, or context menu work: read `input-systems.md`.
- Visual QA or screenshot review: read `verification.md` and `checklist.md`.
- Accessibility/Dynamic Type/focus issue: read `accessibility.md`.

## Defaults

- Keep the iPad app visually and behaviorally related to the iPhone app.
- Do not duplicate the iPhone layout at larger sizes.
- Use platform navigation and windowing before inventing custom chrome.
- Support keyboard and pointer for repeated productivity workflows.
- Verify current iPadOS APIs against Apple docs before adopting new OS-specific chrome or window controls.
